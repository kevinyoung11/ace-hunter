# Ace Hunter 官网设计 Prompt SOP

日期：2026-07-19
状态：待用户书面审核

## 1. 目标

为 Ace Hunter 营销官网提供一套可逐条复制、分阶段执行的设计 Prompt。官网唯一主转化目标是让目标用户安装 Ace Hunter Skill。

本 SOP 只设计营销官网，不建设 Ace Hunter 产品 Web 前端。日报、项目分析、实时观察、关注列表等产品能力仍通过 Skill 交付，必须遵守 `2026-07-19-ace-hunter-skill-first-design.md` 中已经确认的产品边界。

## 2. 使用对象与工具分工

Prompt 包面向使用 Open Design、Hallmark、UI/UX Pro Max 和 Codex 的执行者。

- Open Design：管理画布、设计系统、方案版本和多轮视觉迭代。
- Hallmark：决定宏观页面结构、视觉方向，并检查常见 AI 生成痕迹。
- UI/UX Pro Max：为产品类型推荐字体、颜色、间距、交互和可访问性规则。
- Codex：读取项目事实、审查截图、实现 React/Next.js 页面并验证响应式结果。

工具建议不构成强依赖。若某个 Skill 不可用，执行者必须保留相同的阶段、产出格式和验收门槛，不得跳过设计审查直接写代码。

## 3. Prompt 包结构

最终交付由一个共享主 Brief 和九个阶段 Prompt 构成。

### 00｜项目上下文与总约束

读取 Ace Hunter 产品规格，建立官网可以表达、需要谨慎表达和禁止表达的事实清单。明确官网是安装入口，不是产品控制台。

产出：固定文件 `WEBSITE-BRIEF.md`，供后续所有阶段引用。

### 01｜官网策略与信息架构

定义核心受众、访问者问题、核心价值承诺、页面叙事、区块顺序和安装转化路径。

产出：信息架构、每个区块的任务、CTA 层级、所需事实证据清单。

### 02｜UI/UX Pro Max 设计系统

根据开发者工具、GitHub 项目发现、数据可信度和 Skill-first 产品形态，形成 Ace Hunter 自有视觉系统，不直接复制 Linear、Vercel、GitHub 或其他知名品牌。

产出：项目级 `DESIGN.md`，包括颜色、字体、字号、间距、栅格、组件、图表、动效、响应式、可访问性、素材规则和反模式。

### 03｜Hallmark 三方向探索

基于同一信息架构生成三个本质不同的完整官网方向。差异必须体现在宏观结构、首屏构图、字体策略、视觉资产、内容节奏和交互方式，不能只是更换颜色。

产出：三个 1440px 可比较的完整方向，以及每个方向的设计意图、优势、风险和适用条件。

### 04｜方向评审与选择

按照品牌辨识度、产品理解速度、安装转化能力、可信度、差异化、响应式潜力和实现成本进行评分。不得因视觉冲击力单项突出就自动胜出。

产出：评分矩阵、淘汰理由、推荐方向及需要吸收的其他方向优点。必须由用户确认最终方向后才能继续。

### 05｜完整桌面方案

把选定方向扩展为 1440px 完整页面，覆盖 Hero、问题、工作方式、核心使用场景、样例输出、可信度说明、安装步骤、FAQ 和最终 CTA。区块可根据策略调整，但必须形成从理解到安装的完整路径。

产出：完整桌面设计、真实文案草案、素材清单和交互说明。

### 06｜响应式设计

为 768px 和 390px 单独调整信息优先级、布局、导航、安装命令、图表、代码块和交互，不允许机械缩放桌面页面。

产出：1440、768、390 三种宽度的可审查页面。

### 07｜第一轮视觉审查

审查信息层级、字体比例、留白、栅格、素材真实性、品牌一致性、交互反馈、AI 生成痕迹和三种宽度的一致性。问题必须对应具体区块和修改动作。

产出：P0/P1/P2 问题清单、修改计划、修改后的三尺寸截图和前后对照。

### 08｜第二轮转化审查

审查访问者是否能快速理解产品、区分事实与愿景、找到安装入口、理解安装后的使用方式，并验证 CTA、信任证据、安装命令、移动端操作、键盘使用和降低动效设置。

产出：转化与可访问性问题清单、修订结果，以及进入实现阶段的通过或退回结论。

### 09｜React/Next.js 实现与验收

仅在前两轮审查通过后实现生产页面。实现必须复用项目现有技术栈和组件约定；如果项目尚无前端技术栈，应先提出最小技术方案并取得确认。

产出：生产代码、资源文件、三尺寸截图、基本交互测试、可访问性检查和与设计稿的差异说明。

## 4. 每条 Prompt 的固定组成

每条阶段 Prompt 必须包含以下七部分：

1. 执行位置：说明适合在 Open Design、Codex 或对应 Skill 中运行。
2. 必读输入：列出产品规格、上一阶段产物、截图或现有代码。
3. 任务：限定本阶段要解决的问题。
4. 禁止事项：阻止过早编码、虚构事实、套用模板和跳过审查。
5. 输出格式：给出固定标题、表格或文件结构。
6. 验收门槛：说明什么条件满足后才能进入下一阶段。
7. 停止条件：缺少事实、方向未确认或审查未通过时必须停下并报告，不得自行补全。

## 5. 官网事实边界

Prompt 必须要求执行者以已确认的产品规格为唯一事实源。

可以表达：

- Ace Hunter 是 Skill-first 的 GitHub 项目发现与观察产品。
- 它围绕“今日值得关注”“分析”“观察”“关注”五类自然语言意图工作。
- 它组合 GitHub 数据和系统采集到的 X 讨论。
- 排名使用确定性指标，AI 负责解释和分类。
- 实时观察是 fresh-on-request，目标响应时间为 30 至 90 秒。
- 第一版通过 Skill 使用，不建设产品 Web 前端。

需要谨慎表达：

- 尚未完成实现或验证的能力应使用“规划”“目标”或“设计为”，不能写成已经上线。
- X 数据源、授权、价格和速率限制尚未确定，不能承诺稳定覆盖全部 X 讨论。
- 推荐效果的提升目标需要至少 30 天和 20 个有效日报 cohort 后才能验证。

禁止虚构：

- 用户数量、客户 Logo、媒体评价、融资信息、已获得的 GitHub Stars。
- 未经验证的准确率、节省时间、预测成功率和 Trending 命中案例。
- 不存在的安装命令、包名、仓库地址、用户评价和产品截图。

若安装命令或发布地址尚未确定，设计阶段必须使用清晰标记的占位内容，并在实现前阻塞确认；不能生成看起来真实但无法执行的命令。

## 6. 视觉反模式

三个方向都必须主动避免：

- 紫蓝霓虹渐变作为默认科技感。
- 全页面由同一种圆角卡片组成。
- 无信息意义的 Bento Grid。
- 漂浮光球、网格背景和装饰性粒子堆叠。
- 通用 AI 机器人、火箭、望远镜或抽象 3D 插画。
- 所有文字和区块居中排列。
- 直接模仿 Linear、Vercel、GitHub 或 Stripe。
- 用虚构 Dashboard 截图冒充已经存在的产品界面。
- 为追求动效牺牲安装路径、可读性或降低动效支持。

## 7. 审查标准

每轮审查必须基于 1440、768、390 三种宽度的真实截图，不接受只阅读代码或只检查单一视口。

第一轮重点：

- 五秒内能否识别产品类别和主要价值。
- Hero 是否拥有明确视觉重心。
- 字体是否形成可辨识而稳定的层级。
- 区块节奏是否有变化但不失一致性。
- 素材是否来自真实产品事实或明确的概念表达。
- 是否仍带有模板化 AI SaaS 页面特征。

第二轮重点：

- 主 CTA 是否始终指向安装 Skill。
- 安装步骤是否可复制、可理解、可验证。
- 是否清楚解释安装后可以问什么。
- 是否区分当前能力、设计目标和尚未验证的结果。
- 移动端代码块、CTA 和导航是否可用。
- 是否支持键盘导航、可见焦点、合理对比度和 `prefers-reduced-motion`。

## 8. 交付形式

最终 Prompt 包以 Markdown 形式交付。每一阶段使用独立代码块，允许直接复制。共享主 Brief 放在最前面，后续 Prompt 引用固定的产物文件名，减少重复粘贴产品背景。

Prompt 包不包含虚构的最终安装命令。如果项目在执行时已有经过验证的安装命令，应由 Codex从仓库事实中读取并填入；否则在实现阶段前向用户请求该信息。

## 9. 完成标准

Prompt 包在满足以下条件后视为完成：

- 完整覆盖 00 至 09 十个阶段。
- 所有阶段都有输入、任务、禁止事项、输出格式、验收门槛和停止条件。
- 明确区分营销官网与非目标中的产品 Web 前端。
- 所有产品描述与现有规格一致。
- 不包含未经核验的安装命令或运营数据。
- 能阻止模型在方向选择和两轮审查前进入生产编码。
